## 二、项目背景

- **项目名称**：AI企服外脑（AI企服规划）
- **一句话定位**：「我 + 专家网络」中小企业老板的「AI落地策划师」——全景诊断→落地策划→持续陪跑，帮老板在 AI 时代看清楚整个棋盘。AI 是我的效率引擎，不是我的身份标签。
- **核心产品**：帮中小企业老板看清楚整个棋盘的判断力——方向判断、方案判断、执行判断，通过「全景诊断→落地策划→持续陪跑」交付
- **叙事框架（v3.6 定案）**：AI 是效率引擎（让我一个人做到传统团队的深度），身份定位是「AI落地策划师」而非「AI顾问」；三轨并列：培训/项目/陪跑，陪跑是持续交付策划价值的载体，但客户可以不买陪跑
- **核心模式**：我 + 专家网络（非 OPC，突出个人信任枢纽 + 专家协作网络）
- **核心壁垒**：四懂（懂业务 + 懂精益 + 懂数据 + 懂AI）× 诊断方法论 × AI效率工具
- **当前阶段**：产品定位深度梳理（AI落地 vs AI决策支持的边界确立）+ 品牌物料制作 + Agent Studio v2.0 重构
- **技术支撑**：Agent Studio v2.0 / 五层知识模型 / Flask + SQLite + 纯HTML/JS / 智谱AI GLM-4-flash

### 四层项目框架

> 整个项目按「从本质到执行」分为四层，不同层有不同受众和产出物：

| 层级 | 内容 | 回答的问题 | 受众 |
|------|------|-----------|------|
| **第一层** | 业务本质与定位 | 这个生意为什么成立？ | 所有人 |
| **第二层** | 方法论与工具支撑 | 凭什么能做到？ | 自己、合作方（验证第一层主张） |
| **第三层** | 客户定位与商业打法 | 卖给谁、怎么卖？ | 客户、市场 |
| **第四层** | 政策与资本 | 怎么拿外部资源？ | 政府、园区、投资人 |

## 五、关键术语表

| 术语 | 定义 |
|-----|------|
| **外脑模式** | 「我 + 专家网络」协作模式：以娜娜为个人信任枢纽，按项目制分账的各领域顶级顾问协作网络。对外不称 OPC。 |
| **OPC** | One Person Company（内部术语，对外不使用）。在武汉的定义：20人以下小团队，非字面意义的"一个人"。 |
| **出血点** | 中小企业业务流程中导致持续性损失的环节（信息/流程/经验/决策四种类型） |
| **专家网络** | 以个人为信任枢纽，按项目制分账的各领域顶级顾问协作网络 |
| **孵化园渠道** | B2B2B模式，通过孵化园/企服公司触达入驻企业 |
| **四锚点** | 合规（底线）+ 效率（内部管理效率，AI是落地工具）+ 增长（生存）+ 创新（未来） |
| **开源·节流** | 诊断顶层分流：开源=对外经营（收入侧），节流=内部管理（成本侧），对应财报收入/成本 |
| **螺旋推进** | 8步诊断流程的增强机制：每步可出阶段性判断+置信度，年度循环加深 |
| **置信度** | 🔴高置信（多方印证）/ 🟡中置信（方向对但缺数据）/ ⚪低置信（仅凭描述） |
| **诊断格调** | 不否认客户，视角差异而非对错，从"另一个视角补充"切入而非"纠正" |

### 📦 制品版本管理流程

#### 制品命名规范
```

示例：诊断Agent架构_v2.1.md

主版本号：重大变更（如整体重构、新增核心模块）
次版本号：小幅修改（如修正措辞、补充细节）
```

2. 更新 INDEX.md 中的版本号
3. git commit 时写明"制品名 v旧→v新：变更摘要"
