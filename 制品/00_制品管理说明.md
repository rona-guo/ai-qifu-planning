# 📦 制品版本管理说明

> **一切产出物都存这里 → 本地先整理好 → 推送到 GitHub → AI 从 GitHub 拉取最新版**

---

## 核心原则

```
本地产出  →  git push  →  GitHub（权威源）  →  AI 每次对话从 GitHub 拉取
     ↑                                                      |
     └────────────── 不要直接读本地（可能是旧版本）←─────────┘
```

**🔑 关键规则：AI 读取制品时，必须先从 GitHub 拉取最新内容，而非直接读本地文件。**

执行方式：
```bash
cd C:/Users/huawei/WorkBuddy/2026-06-10-10-28-33/IMA知识库 && git pull origin main
```

---

## 目录结构

```
制品/
├── 00_制品管理说明.md     ← 本文件
├── INDEX.md               ← ⭐ 制品版本索引（单一事实来源）
├── 系统方案/              ← 架构图、API设计、技术方案
├── 产品设计/              ← PRD、原型、用户流程
├── 数据分析/              ← 报表、图表、分析结论
└── [制品名]_vX.Y.md      ← 零散的重要文档
```

---

## 版本索引 INDEX.md

**这是制品版本的「单一事实来源」。AI 必须先查 INDEX.md 才能确定最新版本号。**

### INDEX.md 格式

```markdown
| 制品ID | 制品名称 | 最新版本 | 文件路径 | 状态 | 最后更新 |
|--------|---------|---------|---------|------|---------|
| ART-001 | 诊断Agent架构 | v2.1 | 系统方案/诊断Agent架构_v2.1.md | 🟢已确认 | 06-11 |
```

### 读取制品的正确方式

```
1. git pull origin main（确保本地最新）
2. 读 INDEX.md → 确定最新版本号
3. 读对应的版本文件
```

❌ **错误做法**：不查 INDEX，直接 glob 匹配文件名 → 可能读到旧版本残留

## 版本号规则（语义化）

```
v[主版本].[次版本]

主版本：重大变更（如整体重构、新增核心模块）
次版本：小幅修改（如修正措辞、补充细节、新增小节）
```

| 变更类型 | 版本号 | 示例 |
|---------|--------|------|
| 全新创建 | v1.0 | 诊断Agent架构_v1.0.md |
| 补充几个小节 | v1.1 | 诊断Agent架构_v1.1.md |
| 整体重写 | v2.0 | 诊断Agent架构_v2.0.md |
| 修正措辞/格式 | v2.1 | 诊断Agent架构_v2.1.md |

---

## 制品更新六步流程

```
1. 用户说「更新 XXX 制品」
2. AI: git pull origin main（确保本地最新）
3. AI: 读 INDEX.md → 确认当前版本
4. AI: 读当前版本文件 → 修改内容
5. AI: 升版本号 + 写新文件 + 更新 INDEX.md
6. AI: git add . && git commit -m "制品名 v旧→v新" && git push
```

---

## 旧版本处理

- **保留旧版本文件**（不删除），方便回溯
- INDEX.md 始终指向最新版本
- 旧版本文件顶部标注：「⚠️ 已被 vX.Y 取代，保留仅用于历史参考」

## 命名规范

```
[制品名称]_v[主版本].[次版本].md

正确示例：
  诊断Agent架构_v2.1.md
  BP商业计划书_v3.0.md
  
错误示例：
  诊断Agent架构_final.md        ← 不用 final、latest 等模糊词
  诊断Agent架构_2026-06-11.md   ← 不用日期替代版本号
```
